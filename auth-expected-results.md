# Auth 修改预期效果与前端验证清单

说明：本文用于在按 Todo List 修改 Auth 之前，先明确修改后应该看到什么效果。后续前端验证由人工完成；AI 不主动运行验证。

## 1. 登录/注册配置泄露修复

### 涉及目标

- `/sign-in`
- `/sign-up`

### 预期效果

- 登录页和注册页功能保持不变。
- 页面仍能正常显示登录/注册表单和第三方登录按钮。
- 页面传给前端的配置只包含公开配置，不包含后台密钥、provider secret、API key、R2 key、支付密钥等敏感配置。

### 如何验证

1. 打开 `/sign-in`。
2. 打开 `/sign-up`。
3. 页面正常渲染，无报错。
4. 在浏览器 DevTools 中查看页面数据或搜索关键敏感字段。

### 应该结果

- 页面仍可正常访问。
- 不应在前端页面数据中看到：
  - `google_client_secret`
  - `github_client_secret`
  - `stripe_secret_key`
  - `stripe_signing_secret`
  - `paypal_client_secret`
  - `resend_api_key`
  - `r2_access_key`
  - `r2_secret_key`
  - AI provider API key

---

## 2. 邮箱验证公开开关

### 涉及目标

- `/sign-up`
- 弹窗注册表单

### 预期效果

- 如果后台配置开启 `email_verification_enabled`，注册表单能显示邮箱验证提示。
- 如果后台配置关闭，注册表单不显示邮箱验证提示。
- 该配置可以暴露给前端，因为它只是公开功能开关，不是密钥。

### 如何验证

1. 确认后台邮箱验证开关开启。
2. 打开注册页或注册弹窗。
3. 查看邮箱输入框附近是否出现邮箱验证提示。
4. 关闭邮箱验证开关后重新查看。

### 应该结果

- 开启时显示邮箱验证提示。
- 关闭时不显示邮箱验证提示。
- 不影响注册按钮、邮箱、密码、名称输入。

---

## 3. 右上角登录状态刷新

### 涉及目标

- 站点右上角用户区域
- 登录弹窗
- 登录后 header 状态

### 预期效果

- 登录成功后，右上角应从“登录/注册按钮”变成用户头像、积分、我的作品入口等已登录状态。
- 不需要手动刷新页面才能看到用户状态。
- 当前项目已有的右上角 UI 不应被替换。

### 如何验证

1. 退出登录。
2. 打开首页或任意有右上角用户区域的页面。
3. 点击登录。
4. 使用邮箱或第三方登录成功。
5. 观察右上角状态。

### 应该结果

- 登录成功后右上角展示当前用户信息。
- credits 按钮仍存在。
- “我的作品”入口仍存在。
- 头像菜单样式保持当前项目样式。
- admin 用户仍能看到 admin 入口。
- 不应出现登录后仍显示“登录”按钮的问题。

---

## 4. OAuth popup 登录

### 涉及目标

- Google 登录
- GitHub 登录
- 登录弹窗中的第三方登录按钮

### 预期效果

- 点击 Google/GitHub 登录后，优先打开 OAuth popup 小窗口。
- 用户在 popup 中完成授权。
- 授权完成后 popup 自动关闭或跳回成功页。
- 主页面收到登录成功状态并刷新用户状态。
- 如果浏览器阻止 popup，应 fallback 到原来的跳转登录方式。

### 如何验证

1. 退出登录。
2. 打开登录弹窗。
3. 点击 Google 登录或 GitHub 登录。
4. 观察是否打开 popup。
5. 完成授权。
6. 回到主页面查看右上角用户状态。
7. 可额外测试浏览器阻止 popup 时是否仍能走 redirect 登录。

### 应该结果

- 正常情况下出现 OAuth popup。
- popup 授权完成后主页面进入已登录状态。
- 登录弹窗关闭。
- 右上角显示用户信息。
- 如果 popup 被阻止，仍可通过 redirect 完成登录。
- 不应出现主页面不刷新、popup 不关闭、登录成功但右上角仍未登录的问题。

---

## 5. sign-modal 登录/注册切换

### 涉及目标

- 登录弹窗
- 注册弹窗
- 移动端 Drawer 登录/注册弹窗

### 预期效果

- 登录弹窗底部点击“注册”后，不跳转页面，而是在同一个弹窗中切换到注册表单。
- 注册表单底部点击“登录”后，切回登录表单。
- 桌面端 Dialog 和移动端 Drawer 都能切换。
- 关闭弹窗后，再次打开默认回到登录模式。

### 如何验证

1. 打开登录弹窗。
2. 点击底部“注册”。
3. 查看是否切换到注册表单。
4. 点击注册表单底部“登录”。
5. 查看是否切回登录表单。
6. 关闭弹窗后重新打开。
7. 在移动端宽度下重复上述流程。

### 应该结果

- 登录/注册可在同一个弹窗内切换。
- 关闭后重新打开默认显示登录。
- Dialog / Drawer 布局不变形。
- 原 `/sign-up` 页面仍可访问。

---

## 6. sign-up-form 注册表单

### 涉及目标

- 弹窗内注册表单

### 预期效果

- 弹窗注册表单包含：
  - name
  - email
  - password
- 点击注册时会调用注册流程。
- 如果开启邮箱验证，注册后跳转或显示验证邮箱提示。
- 如果无需邮箱验证，注册后更新用户状态并关闭弹窗。

### 如何验证

1. 打开登录弹窗。
2. 切换到注册。
3. 不填写内容点击注册。
4. 填写 name/email/password 后提交。
5. 分别在邮箱验证开启和关闭时测试。

### 应该结果

- 未填写完整时显示“需要姓名、邮箱和密码”的提示。
- 注册失败时显示注册失败提示。
- 邮箱验证开启时，提示用户验证邮箱。
- 邮箱验证关闭时，注册成功后弹窗关闭，右上角进入登录状态。

## 7. 文案 key

### 涉及目标

- 登录弹窗
- 注册弹窗
- 注册页

### 预期效果

新增或使用的 Auth 文案不应缺失。

### 如何验证

1. 切换英文语言。
2. 打开登录/注册弹窗。
3. 触发空表单错误、注册失败错误。
4. 切换中文语言重复测试。

### 应该结果

- 英文和中文都不显示 raw key。
- 不应出现类似 `email_password_name_required`、`sign_up_failed` 这样的未翻译 key。
- 不修改任何 landing、首页、pricing、showcases、blog、docs、法务、meta 等 SEO 文案。

---

## 8. proxy / middleware

### 涉及目标

- locale 路由
- 登录跳转
- Cloudflare / Next 入口约定

### 预期效果

- 不新增 `src/middleware.ts`。
- 继续使用当前 `src/proxy.ts`。
- 登录、注册、locale 切换、普通页面访问不受影响。

### 如何验证

1. 打开默认语言首页。
2. 切换语言。
3. 打开 `/sign-in`、`/sign-up`。
4. 登录后跳转 callbackUrl。
5. 访问普通页面。

### 应该结果

- 不出现 Not found。
- locale 前缀正常。
- 登录跳转正常。
- 不因缺少 middleware 产生路由异常。

---

## 9. 不应该发生的变化

以下变化如果出现，说明改动超出预期：

- 右上角用户菜单 UI 被替换。
- credits 按钮消失。
- “我的作品”入口消失。
- admin 入口消失。
- landing / 首页 / pricing / showcases SEO 文案变化。
- 登录风控被移除。
- 上传、支付、AI 生成链路受到影响。
- 新增 `src/middleware.ts`。
- 整体覆盖 `auth/config.ts`。
