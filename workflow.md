# AI 编程工作流：先计划，后实现

> 来源整理：公众号文章《用了 9 个月 AI 编程后，我只推荐这一套工作流——别让 AI 直接写代码》  
> 原文链接：https://mp.weixin.qq.com/s/TjOWmGd1e8QdNDAisec7GQ  
> 参考原文提到的 Boris Tane 工作流，并整理为可直接执行的团队/个人流程。

## 一句话原则

**不要一上来让 AI 写代码。先让 AI 深入理解代码库，再产出书面计划；人审核并批注计划，确认后才进入实现。**

这套流程的重点是把“思考”和“打字”分开：

- AI 负责阅读、归纳、拆解、执行。
- 人负责判断方向、审查计划、控制范围、决定取舍。
- `research.md` 和 `plan.md` 是整个过程的共享上下文和审查面。

---

## 总流程

```text
Research → Plan → Annotate（循环 1-6 轮）→ Todo List → Implement → Feedback & Iterate
```

| 阶段 | 目标 | 产物 | 人的职责 | AI 的职责 |
|---|---|---|---|---|
| Research | 先读懂现有系统 | `research.md` | 审查理解是否正确 | 深读代码并写研究报告 |
| Plan | 先出方案，不写代码 | `plan.md` | 判断方案是否可行 | 基于真实代码写实现计划 |
| Annotate | 通过批注校准方案 | 更新后的 `plan.md` | 加约束、纠错、砍范围 | 按批注修改计划 |
| Todo List | 拆成可执行任务 | 计划中的任务清单 | 确认任务粒度 | 生成阶段和任务列表 |
| Implement | 按计划一次性实现 | 代码 + 已勾选任务 | 监督执行 | 实现、标记进度、修复类型问题 |
| Feedback | 小步纠偏 | 后续补丁 | 快速指出偏差 | 按反馈调整 |

---

## 阶段 1：Research —— 让 AI 先读懂代码

### 目标

在任何实现前，先让 AI 深入阅读相关目录、模块、流程或问题区域，形成一份可审查的研究报告。

### 为什么必须有 `research.md`

聊天里的口头总结很容易丢失、难审查，也难复用。`research.md` 可以作为：

- 代码理解的持久化记录；
- 后续计划的输入基础；
- 人类审查 AI 是否真的理解系统的窗口；
- 避免 AI 在错误假设上继续设计和实现的防线。

### 推荐指令模板

```text
请深入阅读 [目录/模块/文件/业务流程]，理解它的职责、调用链、边界条件、数据结构、已有约定和潜在风险。

完成后，把你的发现写入 research.md，内容至少包括：
1. 模块整体职责
2. 关键文件和函数说明
3. 数据流 / 调用链
4. 现有设计约定
5. 可能的风险、坑点或 bug
6. 后续改动时必须注意的约束

只做研究和写文档，不要修改代码。
```

### 审查清单

在进入下一步前，人工检查 `research.md`：

- [ ] AI 是否读了正确的目录和文件？
- [ ] 是否理解了真实调用链，而不是只看表面函数名？
- [ ] 是否遗漏了缓存、权限、异步任务、数据库迁移、类型约束等关键因素？
- [ ] 是否把项目已有约定写清楚？
- [ ] 是否有明显误解？如有，先纠正研究报告。

## 阶段 2：Plan —— 让 AI 写计划，不写代码

### 目标

在研究通过后，让 AI 基于真实代码库写一份详细实现计划 `plan.md`。

### 计划必须包含

- 要实现的业务目标；
- 需要修改的文件路径；
- 每个文件的修改意图；
- 核心代码结构或伪代码；
- 数据结构 / API / 类型变化；
- 迁移、兼容性、权限、缓存、错误处理等注意事项；
- 风险与权衡；
- 暂不实现的内容。

### 推荐指令模板

```text
基于 research.md 和当前代码库，我要实现：[功能名称和业务目标]。

请写一份详细的 plan.md，说明如何实现。要求：
1. 先基于真实源文件分析，不要凭空设计；
2. 列出会修改的文件路径；
3. 说明每处修改的原因和方式；
4. 必要时给出关键代码片段或伪代码；
5. 写清楚风险、取舍和不做的内容；
6. 只写计划，不要实现。
```

### 如果有参考实现

把参考代码或参考页面直接提供给 AI：

```text
这里有一个参考实现：[粘贴代码/说明路径]。
请不要照搬，而是分析它的设计方式，并写入 plan.md：我们应该如何在当前项目中采用类似方案。
不要实现。
```

## 阶段 3：Annotate —— 人类批注计划

### 目标

打开 `plan.md`，直接在文档里批注、改写、否决或补充。不要急着实现。

### 可以批注什么

- AI 的错误假设；
- 不符合业务的技术方案；
- 不允许变更的接口；
- 应该删除的范围；
- 必须使用的库、命令或项目约定；
- 产品、运营、性能、权限等 AI 不知道的背景；
- 更合适的已有模块或参考实现。

### 批注方式示例

```md
<!-- 批注：这里不能新增 raw SQL，迁移必须走项目已有 migration 工具。 -->

<!-- 批注：这个接口不应该改签名，下游调用很多；请让调用方适配。 -->

<!-- 批注：删除缓存设计，本阶段不做缓存，避免扩大范围。 -->

<!-- 批注：这里应复用 users table 的表格样式，包括分页、行高和表头结构。 -->
```

### 让 AI 处理批注

```text
我已经在 plan.md 里添加了批注。
请逐条处理这些批注，并更新 plan.md。
只更新计划，不要实现代码。
```

### 循环规则

- 通常循环 1-6 轮；
- 每一轮只允许 AI 更新计划；
- 直到人类认为计划足够明确、范围足够稳定、风险已被处理；
- 计划没确认前，不进入实现。

## 阶段 4：Todo List —— 生成可追踪任务清单

### 目标

在 `plan.md` 中加入细粒度任务列表，让实现过程有唯一进度来源。

### 推荐指令模板

```text
请在 plan.md 中添加详细 Todo List。
要求：
1. 按阶段拆分；
2. 每个任务足够具体，可完成后勾选；
3. 包含代码修改、测试/类型检查、文档更新等必要事项；
4. 不要实现代码。
```

### Todo List 示例结构

```md
## Todo List

### Phase 1：数据结构调整
- [ ] 更新类型定义
- [ ] 调整数据读取逻辑
- [ ] 处理兼容旧数据的分支

### Phase 2：接口实现
- [ ] 修改接口参数解析
- [ ] 增加错误处理
- [ ] 保持旧调用方兼容

### Phase 3：前端接入
- [ ] 更新页面状态管理
- [ ] 复用现有组件样式
- [ ] 处理 loading / empty / error 状态

### Phase 4：验证
- [ ] 运行类型检查
- [ ] 验证关键路径
- [ ] 标记所有任务完成状态
```

---

## 阶段 5：Implement —— 计划确认后再实现

### 目标

当 `plan.md` 已经被充分审核后，让 AI 按计划完整实现，并持续更新任务状态。

### 推荐指令模板

```text
现在按 plan.md 完整实现。

要求：
1. 严格按照 plan.md 执行，不要擅自扩大范围；
2. 每完成一个任务或阶段，就在 plan.md 中勾选；
3. 不要中途停下来询问，除非遇到真实阻塞；
4. 不要添加无意义注释；
5. 遵守项目现有类型、命名、样式和架构约定；
6. 按计划运行必要的类型检查或验证；
7. 完成所有任务后，总结改动。
```

### 实现阶段注意事项

- 实现阶段应该尽量机械化，不再做重大设计决策；
- 如果发现计划错误，应暂停修改计划，而不是边写边随意改方向；
- 任务进度以 `plan.md` 的 Todo List 为准；
- 人类主要做监督和小范围纠偏。

## 阶段 6：Feedback & Iterate —— 实现中的快速纠偏

### 目标

AI 执行过程中，人类用短反馈纠正偏差。

### 反馈原则

- 越具体越好；
- 能引用现有代码就引用现有代码；
- UI 问题优先用截图或明确参考页面；
- 方向错了就回滚并缩小范围，不要在错误方向上继续修补。

### 反馈模板

```text
你漏实现了 [具体函数/文件/任务]，请补上，并更新 plan.md 的任务状态。
```

```text
这个页面应该复用 [已有页面/组件] 的结构和样式，不要重新设计。
```

```text
这次改动超出了 plan.md 范围，请撤回 [具体部分]，只保留 [目标改动]。
```

```text
我已经回滚了之前的改动。现在只做一件事：[缩小后的目标]。不要改其他内容。
```

## 控制权规则

### 1. 人决定方向，AI 执行任务

AI 可以提出方案，但最终由人决定：

- 哪些问题值得解决；
- 哪些范围要砍掉；
- 哪些接口不能动；
- 哪些技术方案符合团队习惯；
- 什么程度算“够好”。

### 2. 防止范围蔓延

当计划里出现锦上添花内容时，直接删除或标记为不做：

```text
从 plan.md 中删除 [功能/模块]，本轮不实现。
```

### 3. 保护现有接口

```text
以下函数/API 的签名不能变：[列表]。
如果需要适配，请修改调用方，不要修改公共接口。
```

### 4. 指定技术选择

```text
这里必须使用项目已有的 [库/工具/组件/命令]，不要自定义实现。
```

### 5. 多问题逐项决策

```text
针对你列出的几个问题：
1. 第一个按简单方案处理；
2. 第二个暂不处理；
3. 第三个提取成独立函数；
4. 其余不做，避免增加复杂度。
请更新 plan.md，不要实现。
```

## 推荐文件结构

```text
项目根目录/
├── research.md   # AI 对现有系统的研究报告
├── plan.md       # 经人类批注和确认的实现计划
└── 源代码文件...
```

如果项目较大，也可以放到任务目录：

```text
docs/ai-plans/2026-05-feature-name/
├── research.md
└── plan.md
```

## 每次任务的执行模板

### Step 1：研究

```text
请深入研究 [范围]，输出 research.md。只研究，不改代码。
```

### Step 2：计划

```text
基于 research.md，为 [目标] 写 plan.md。只写计划，不改代码。
```

### Step 3：批注

人工打开 `plan.md`，直接添加批注。

### Step 4：更新计划

```text
我已在 plan.md 添加批注。请处理所有批注并更新计划，不要实现。
```

### Step 5：生成任务清单

```text
请在 plan.md 添加分阶段 Todo List，不要实现。
```

### Step 6：实现

```text
按 plan.md 完整实现。完成任务后勾选 Todo。除非真实阻塞，否则不要中途停下。
```

### Step 7：反馈修正

```text
请修正 [具体偏差]，不要扩大范围，并更新 plan.md 状态。
```

## 适用场景

特别适合：

- 中大型功能开发；
- 复杂 bug 排查；
- 涉及多文件、多模块的重构；
- 需要遵守既有架构和业务约定的项目；
- 长时间 AI 编程会话；
- 需要团队审查和沉淀记录的任务。

不一定适合：

- 一两行的小改动；
- 明确且无依赖的文本替换；
- 完全独立的小脚本；
- 没有现有代码上下文的原型验证。

---

## 最终检查清单

开始实现前：

- [ ] `research.md` 已完成并审查；
- [ ] `plan.md` 已基于真实代码生成；
- [ ] 人类已完成批注；
- [ ] AI 已处理所有批注；
- [ ] Todo List 已拆分；
- [ ] 范围已明确，暂不做内容已写清；
- [ ] 不可变更的接口和约定已写入计划；
- [ ] 验证方式已写入计划。

实现完成后：

- [ ] Todo List 全部勾选；
- [ ] 没有超出计划范围的改动；
- [ ] 类型检查 / 测试 / 手动验证按计划完成；
- [ ] 关键风险已处理；
- [ ] 最终改动已总结。

---

## 核心结论

这套工作流不是追求“让 AI 更自由”，而是让 AI 在清晰边界内高效执行：

1. **Research** 防止 AI 不理解项目就乱改；
2. **Plan** 防止 AI 直接做错方向；
3. **Annotate** 把人的业务判断和工程取舍注入方案；
4. **Todo List** 让长任务可追踪；
5. **Implement** 让 AI 在决策完成后连续执行；
6. **Feedback** 用短指令持续纠偏。

真正的价值在于：**用文档把上下文、决策和执行过程固定下来，让人始终掌控方向，让 AI 专注完成执行。**
